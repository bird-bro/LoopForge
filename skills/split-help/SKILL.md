# Split Monolithic CLAUDE.md into Agent Workspaces

Take a monolithic CLAUDE.md generated by `/init` (which mixes backend, frontend, mobile into one file) and intelligently split it into per-directory Agent CLAUDE.md files.

## Why This Matters

`/init` treats the entire project as one workspace. It puts Java/MyBatis patterns next to Vue/Element Plus patterns. When AI reads this, it gets confused — Java DI patterns while fixing a CSS bug. Each Agent should only see its own domain.

## When to Use

- After running `/init` on a multi-stack monorepo
- When a single CLAUDE.md has 200+ lines covering multiple tech stacks
- When adding a new technology layer to an existing project

## What You Need Before Starting

- The monolithic CLAUDE.md generated by `/init` (or equivalent)
- Knowledge of which code directories map to which tech stacks

---

## Phase 1: Discovery

### Step 1: Read the Monolith

Read the root CLAUDE.md completely. Identify every section and classify it:

| Content Type | Belongs In |
|:---|:---|
| Business context / system description | Root CLAUDE.md (nav) + `openspec/project.md` |
| Build commands (Maven/Gradle) | Backend Agent CLAUDE.md |
| Build commands (pnpm/npm/yarn) | Frontend Agent CLAUDE.md |
| Backend module structure | Backend Agent CLAUDE.md |
| Frontend directory structure | Frontend Agent CLAUDE.md |
| Java/MyBatis/Spring patterns | Backend Agent CLAUDE.md |
| Vue/React/Element Plus patterns | Frontend Agent CLAUDE.md |
| Response format / Error codes / Status codes | Both (backend + frontend look at these differently) |
| Naming conventions (shared) | If truly universal → keep a minimal version in each Agent. If domain-specific → one Agent only |
| API paths | `specs/api-contract.md` (not Agent files!) |

### Step 2: Confirm with User

Before generating files, present your findings. For each identified stack, ask:

```
I found these tech stacks in the monolith:

1. Backend: [language/framework] → code in [directory]
   Sections to move: [list of sections]

2. Frontend Web: [language/framework] → code in [directory]
   Sections to move: [list of sections]

3. Frontend Mobile: [language/framework] → code in [directory]
   Sections to move: [list of sections]

Does this look correct? Any stacks I missed or misidentified?
```

Wait for user confirmation before proceeding.

## Phase 2: Generate Agent CLAUDE.md Files

For each tech stack, generate a file at `{code-directory}/CLAUDE.md`. Follow this exact structure:

```markdown

You are a **[Stack] [Role] Agent**. Your scope: [specific responsibilities].
**NEVER generate [opposite-domain] code**. **NEVER modify `[relative-path-to-specs/]`**.
**This file covers [this domain] only. [Other domain] is at `[other-agent-path]`.**

- **System**: [one-line business description]
- **Stack**: [languages, frameworks, versions]

- **Config**: [config system] — only if relevant

1. **Read specs first**: Always read `[specs-path]` — API contract, data model, error codes

## Superpowers Workflow

All development must follow this command sequence:

```bash
# 1. Brainstorm — analyze specs and generate approach
/superpowers:brainstorm "Based on [specs-path] contracts, plan [feature]"

# 2. Write plan — decompose into 2-5 minute tasks
/superpowers:writing-plans

# 3. Execute plan — implement in batches with review checkpoints
/superpowers:executing-plans

# 4. Code review — after each batch
/superpowers:code-review --file [file-pattern]

# 5. Final verification — before claiming completion
/superpowers:verification-before-completion
```

### TDD is Mandatory (enforced by Superpowers)

```bash
/superpowers:workflow activate tdd
```

When TDD is active, the AI **cannot proceed to implementation** until tests are written and reviewed.

### Content Extraction Rules

When moving content from the monolith to Agent files:

1. **Code examples stay**: Java code blocks → backend Agent. Vue/SCSS code blocks → frontend Agent.
2. **API paths go to `specs/api-contract.md`**: Do not put endpoint lists in Agent files. If the monolith has inline API docs, extract them to specs.
3. **Response format / Error codes**: These are shared. Include a minimal reference in both Agents, but the authoritative definition lives in `specs/`.
4. **Build commands**: Split by tool. `mvn *` → backend. `pnpm *` / `npm *` → frontend.
5. **Naming conventions**: If a convention applies to both stacks (e.g., "use kebab-case for file names"), it goes in `.claude/rules/` or `specs/`. If domain-specific (e.g., "Entity class naming: {Domain}Entity"), it goes in the relevant Agent file.
6. **Business context**: Goes in root CLAUDE.md (nav hub), NOT duplicated in every Agent file. Each Agent gets a one-line description under "System".

### Handle Edge Cases

**If there is only one tech stack:** Don't split. Add Role + NEVER + Superpowers + TDD sections to the existing file. That's all that's needed.

**If content is already split:** Verify each Agent file has all required sections (Role, Superpowers Workflow, TDD, Build Commands). Add any that are missing. Do not rewrite existing content.

**If a section doesn't map cleanly:** Default to creating a `.claude/rules/` file or adding it to `specs/`. Err on the side of keeping Agent files lean.

**If the monolith has mobile + desktop frontends:** Create separate Agent files. A Vue 3 + Vant mobile file is very different from a Vue 3 + Element Plus desktop file — different component libraries, different layout patterns, different build configurations.

## Phase 3: Rewrite Root CLAUDE.md

After all Agent files are generated, rewrite the root CLAUDE.md as a navigation hub. Content:

```markdown
# [Project Name]

> Structured per OpenSpec + Superpowers + Harness. Launch from subdirectories for development.

## Project Map
[Directory tree showing: root CLAUDE.md, openspec/, specs/, .claude/, each agent directory]

## Business Context
[1-3 sentences from the original CLAUDE.md]

## Tech Stack
[One-line table: Layer | Tech]

## Development Workflow

### New Feature
1. Write spec in `openspec/changes/`
2. Launch backend Agent: `cd [backend-dir] && claude`
3. Launch frontend Agent (parallel): `cd [frontend-dir] && claude`

### AI Coding Rules
- Spec first — read `specs/` before writing code
- TDD — `/superpowers:workflow activate tdd`
- No cross-domain — backend writes zero frontend code, vice versa
- Never modify existing methods — use overloading

### Harness Session Commands
`/resume` — Resume previous session
`/branch` — Branch current session for parallel exploration
`/rewind` — Rewind session to prior state

## Build & Test
[Minimal: 3-5 commands covering all stacks. Detailed commands are in Agent files.]
```

**Sizing rule: Root CLAUDE.md ≤ 120 lines.** If it exceeds, move sections to `openspec/project.md` or Agent files.

---

## Phase 4: Verify

After generating all files, verify:

1. **No content lost**: Every code example, convention, and command from the original CLAUDE.md is in exactly one file.
2. **No duplication**: The same rule does not appear in two files Claude Code would both load.
3. **All required sections present**: Each Agent file has: Role, Project Overview, Before You Code, Module Structure, Coding Standards, Superpowers Workflow, TDD, Build Commands.
4. **Cross-domain prohibition explicit**: Every Agent file states what it MUST NOT do.
5. **Path references correct**: Spec paths (`../specs/`) are relative to each Agent's working directory.
6. **Root CLAUDE.md is a hub**: ≤ 120 lines, no agent instructions, no coding standards.

---

## Anti-Patterns

| Anti-pattern | Fix |
|:---|:---|
| Business context in every Agent file | One-line "System" only; full context in root |
| API paths in Agent files | Extract to `specs/api-contract.md` |
| Duplicate build commands | Each command in exactly one Agent file |
| Mobile + desktop in one Agent file | Separate them — different UI paradigms |
| "This file provides guidance" as first line | Must be "## Role: You are a [Stack] [Role] Agent" |
| No TDD section | Always add Superpowers Workflow + TDD section |
| Root CLAUDE.md > 120 lines | Move content to Agent files or `openspec/project.md` |
